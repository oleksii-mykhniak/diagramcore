# DiagramCore — план реалізації

> Виконуваний план для агентів. Кожен крок має **Acceptance Criteria (AC)** — механічно перевірювані умови.
> Правила виконання: кроки виконуються строго послідовно в межах фази; крок вважається завершеним лише коли ВСІ його AC проходять; після кожного кроку — commit з повідомленням `phase<N>-step<M>: <опис>`.

## Загальні рішення (зафіксовано, не переглядати без згоди власника)

- **Формат ядра:** YAML (`*.dc.yaml`) — мовно-незалежне джерело правди.
- **CLI:** Go (модуль `github.com/oleksii94/diagramcore`, бінарник `dc`). Go ≥ 1.22.
- **Рендеринг фази 3:** нативно через Go-бібліотеку D2 (`oss.terrastruct.com/d2`) + текстова транспіляція в Mermaid.
- **Layout зберігається окремо** від ядра: `<name>.layout.json`.
- **Flow ходить тільки по існуючих ребрах** — крок без відповідного link = помилка валідації.
- **Web-редактор (фаза 5):** TypeScript, окремий застосунок; валідація через Go→WASM.
- Мова коду/ідентифікаторів — англійська; повідомлення CLI — англійська; приклади діаграм можуть бути українською.

## Структура репозиторію (цільова)

```
diagramcore/
├── PLAN.md
├── go.mod
├── cmd/dc/main.go            # CLI entrypoint (cobra або stdlib flag)
├── internal/
│   ├── model/                # структури ядра: Diagram, Node, Link, Flow
│   ├── parser/               # YAML → model
│   ├── validate/             # правила валідації
│   ├── context/              # генератор AI-контексту
│   ├── transpile/            # → D2, → Mermaid
│   └── render/               # виклик D2-бібліотеки, SVG/PNG, steps
├── schema/
│   └── diagramcore.schema.json   # JSON Schema формату (для редакторів/CI)
├── examples/
│   ├── auth-system.dc.yaml
│   ├── oauth-detail.dc.yaml
│   └── <ще 1-2 реальні діаграми>
├── testdata/                 # валідні та невалідні фікстури для тестів
└── web/                      # фаза 5 (TypeScript)
```

# Фаза 0 — Spike: формат на реальних прикладах

**Мета:** зафіксувати v0 формату YAML до написання коду.

### Крок 0.1 — Ініціалізація репозиторію
- `git init`, `go mod init github.com/oleksii94/diagramcore`, `.gitignore` (Go + node), скопіювати концепт-документ у `docs/concept.md`, цей план у корінь.
- **AC:**
  - [ ] `git log` містить initial commit.
  - [ ] `go build ./...` завершується без помилок (порожній модуль — ок).

### Крок 0.2 — Специфікація формату v0
Створити `docs/format.md`, який визначає структуру `*.dc.yaml`:

```yaml
diagram:
  title: "Система авторизації"
  purpose: "..."
  audience: students
  version: "0.1"

nodes:
  - id: User            # унікальний, [A-Za-z][A-Za-z0-9_]*
    type: actor         # з дозволеного набору типів
    label: "Користувач" # опційно, default = id
    description: "..."
    ai_context: "..."   # опційно
    tags: [auth]        # опційно
    details: ./oauth-detail.dc.yaml   # опційно, референс на під-діаграму

links:
  - from: User
    to: Gateway
    type: request       # request|call|query|event|dataflow|inherits|contains
    label: "HTTPS запит"
    directed: true      # default true

flows:
  - name: "Успішна авторизація"
    steps:
      - from: User
        to: Gateway
        note: "Надсилає запит з Bearer токеном"
      - branch:                      # умовна гілка
          condition: "токен валідний"
          then: [ {from: AuthService, to: DB, note: "..."} ]
          else: [ {from: AuthService, to: ErrorHandler, note: "..."} ]
```

Базові типи вузлів v0: `actor, service, storage, queue, external, component`. Розширення типів — поле `custom_types` на рівні diagram (описати в format.md, реалізація у фазі 1).
- **AC:**
  - [ ] `docs/format.md` описує всі секції (diagram, nodes, links, flows), кожне поле має тип, обов'язковість, default.
  - [ ] Описано семантику flow: крок валідний лише якщо існує link між from/to (для directed — у правильному напрямку; зворотний хід по directed-ребру дозволений і трактується як "відповідь/response").
  - [ ] Описано формат референсу на під-діаграму та правило: шлях відносний до поточного файлу.

### Крок 0.3 — Три реальні приклади
Написати вручну `examples/auth-system.dc.yaml` (з концепту, включно з обома flow), `examples/oauth-detail.dc.yaml` (під-діаграма) і ще один приклад іншого домену (напр. обробка платежу з branch).

  - [ ] Приклади відповідають `docs/format.md` (ручна перевірка агентом: жодного поля, не описаного у форматі).

# Фаза 1 — Ядро-бібліотека та `dc validate`

**Мета:** CLI, що парсить і семантично валідує діаграми. JSON Schema ловить структуру; Go-код ловить семантику.

### Крок 1.1 — Модель і парсер
`internal/model`: структури `Diagram, Node, Link, Flow, Step, Branch`. `internal/parser`: `Parse(path) (*Diagram, error)` через `gopkg.in/yaml.v3`, зі збереженням номерів рядків для повідомлень про помилки (yaml.Node).
- **AC:**
  - [ ] Unit-тести: парсинг усіх файлів з `examples/` повертає модель без помилок; поля вибірково звірені (кількість вузлів, назви flows).
  - [ ] Парсинг файлу з синтаксичною помилкою YAML повертає помилку з номером рядка.
  - [ ] `go test ./... && go vet ./...` — зелені.

### Крок 1.2 — Семантична валідація
`internal/validate` з правилами (кожне — окрема функція, окремий код помилки `DCxxx`):
- `DC001` дублікат node id; `DC002` link на неіснуючий вузол; `DC003` невідомий тип вузла/зв'язку (з урахуванням custom_types); `DC004` flow-крок між вузлами без link; `DC005` flow-крок на неіснуючий вузол; `DC006` `details`-файл не існує або сам не парситься; `DC007` порожній flow; `DC008` branch без then.
Результат — список помилок (усі одразу, не fail-fast), кожна з file:line.
Обхід `details`: валідатор рекурсивно перевіряє під-діаграми з множиною відвіданих файлів (канонічний абсолютний шлях) — кожен файл валідується рівно один раз; циклічні референси не є помилкою і не спричиняють нескінченний обхід (див. `docs/format.md`).
- **AC:**
  - [ ] `testdata/` містить ≥8 невалідних фікстур — мінімум по одній на кожен код помилки; unit-тест на кожну перевіряє, що повертається саме очікуваний код.
  - [ ] Всі `examples/` проходять валідацію без помилок.
  - [ ] Файл із двома незалежними помилками повертає обидві.
  - [ ] Фікстура з циклічним `details` (A → B → A): валідація завершується (не зависає), exit 0, кожен файл перевірено один раз; помилка всередині B репортиться рівно один раз.

### Крок 1.3 — CLI `dc validate`
`cmd/dc`: команда `dc validate <files...>` (підтримка glob). Вивід: людиночитний список помилок `file:line [DCxxx] message`; прапорець `--json` для машинного виводу. Exit code 0 = валідно, 1 = помилки валідації, 2 = помилка виконання (файл не знайдено тощо).
- **AC:**
  - [ ] `go build -o dc ./cmd/dc` збирає бінарник.
  - [ ] `./dc validate examples/*.dc.yaml` → exit 0, вивід містить summary (скільки файлів OK).
  - [ ] `./dc validate testdata/invalid_flow_no_link.dc.yaml` → exit 1, вивід містить `DC004` і номер рядка.
  - [ ] `./dc validate nonexistent.yaml` → exit 2.
  - [ ] `./dc validate --json ...` видає валідний JSON (перевірити `| python3 -m json.tool`).

### Крок 1.4 — CI репозиторію
GitHub Actions: на PR/push — `go build`, `go test`, `go vet`, `./dc validate examples/*.dc.yaml`.
- **AC:**
  - [ ] `.github/workflows/ci.yml` існує і локальна симуляція кроків (ті самі команди в shell) проходить.

# Фаза 2 — Генератор AI-контексту

**Мета:** `dc context file.dc.yaml` → markdown-опис для вставки в AI-чат.

### Крок 2.1 — Генератор
`internal/context`: з валідної моделі генерує markdown: заголовок+purpose+audience → секція Components (тип, label, description, ai_context) → секція Links (людською мовою, з label) → по секції на кожен Flow (нумеровані кроки з note, branch як відступні під-списки) → якщо є `details`-референси, дописати "Node X has a detailed sub-diagram: <path>" і за прапорцем `--deep` інлайнити під-діаграму. `--deep` обходить референси з множиною відвіданих файлів: кожна під-діаграма інлайниться один раз, повторний/циклічний референс дає лише рядок-згадку без інлайну.

  - [ ] Вивід для `auth-system` містить всі 5 вузлів, всі links, обидва flows, і жодного вузла/зв'язку, якого немає у джерелі.

# Фаза 3 — Рендеринг через D2/Mermaid + CI-інтеграція

### Крок 3.4 — Flow: статичне підсвічування

### Крок 3.6 — Анімований SVG (D2 steps)

# Фаза 5 — Web-редактор (TypeScript)

**Мета:** інтерактив: flow-плеєр, drag&drop, кілька view. Окремий застосунок у `web/`, споживає ті самі файли.

- **AC:**
  - [ ] `web/public/dc.wasm` збирається make-таргетом; JS-тест (node + wasm_exec.js) валідує рядок з examples → 0 помилок, зіпсований рядок → DC004.

### Крок 5.2 — Скелет застосунку
Vite + React + TypeScript у `web/`. Відкриття `.dc.yaml` (file input / drag файлу), парсинг (js-yaml) + валідація через WASM, рендер графу: elkjs для layout + SVG-компоненти.
- **AC:**
  - [ ] `npm run build` без помилок; `npm test` (vitest) зелений.
  - [ ] Відкриття `auth-system.dc.yaml` показує всі 5 вузлів і всі ребра (playwright-тест).

### Крок 5.3 — Drag & drop → layout.json
Перетягування вузла змінює лише позиції; кнопка "Export layout" скачує `<name>.layout.json` у форматі фази 4; імпорт layout-файлу відновлює позиції.
- **AC:**
  - [ ] Playwright: перетягнути вузол → експортований JSON містить нові координати; ядро-YAML не змінюється (застосунок його взагалі не переписує).
  - [ ] Експортований layout, підкладений до CLI `dc render`, дає ті самі позиції (крос-перевірка з фазою 4).

### Крок 5.4 — Flow-плеєр
Вибір flow зі списку → контроли: prev/next/autoplay. Активний крок: підсвічене ребро + анімація руху маркера по ребру (`<animateMotion>`), панель з `note` поточного кроку. Branch відображається як вибір гілки користувачем.
- **AC:**
  - [ ] Playwright: вибір flow і 3× "next" → підсвічено перші 3 кроки, панель показує note 3-го кроку.

### Крок 5.5 — Drill-down навігація по `details`
Вузол із `details`-референсом має видимий індикатор (іконка ⊞ / стиль); подвійний клік (або кнопка в панелі вузла) відкриває під-діаграму в тому ж редакторі. Стек переходів + breadcrumbs угорі ("auth-system › oauth-detail"), клік по крихті повертає на відповідний рівень зі збереженням стану (позиції, вибраний flow). Під-діаграма читається відносно шляху батьківського файлу; помилка завантаження (файл відсутній/невалідний) показується як нефатальне повідомлення, без падіння застосунку.
- **AC:**
  - [ ] Playwright: відкрити `auth-system.dc.yaml` → вузол з `details` має індикатор, вузли без — ні.
  - [ ] Playwright: подвійний клік по вузлу з `details` → рендеряться вузли `oauth-detail`, breadcrumbs показує обидва рівні; клік по "auth-system" повертає верхню схему з тими самими позиціями вузлів.
  - [ ] Битий `details`-шлях → повідомлення про помилку, редактор лишається на поточній схемі.

1. Прогнати всі AC кроку як чекліст; AC з командами виконувати буквально і звіряти exit code/вивід.
