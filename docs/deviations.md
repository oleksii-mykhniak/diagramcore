## Крок 4.2 — Рендер з урахуванням layout
- Причина: план передбачав використання нативних D2 reserved keywords
  `top`/`left` для фіксації позицій вузлів ("D2 підтримує фіксовані
  позиції через top/left — використати їх"). Дослідження бібліотеки
  `oss.terrastruct.com/d2` (v0.7.1) показало, що `top`/`left` парсяться
  компілятором (`d2graph.Attributes.Top/Left`), але **жоден із
  бандлованих OSS layout-движків (`d2dagrelayout`, `d2elklayout`) їх не
  консумує** — перевірка підтримки (`d2plugin.FeatureSupportCheck`,
  `TOP_LEFT` feature) стосується лише зовнішньої plugin-системи D2 CLI
  (напр. пропрієтарних движків Terrastruct), яку `d2lib.Compile` не
  викликає. Тобто підхід із плану технічно нездійсненний з доступною
  версією бібліотеки.
- Рішення: замість покладання на D2-мову, `internal/render` вручну
  відтворює внутрішній pipeline `d2lib.Compile` (`d2compiler.Compile` →
  `Graph.ApplyTheme` → `Graph.SetDimensions` → `d2layouts.LayoutNested` →
  `d2exporter.Export`) і додає хук між layout та export: після
  автолейауту позиції вузлів, чиї id є в `Options.Positions`, напряму
  перезаписуються (`obj.TopLeft = geo.NewPoint(x, y)`) перед експортом.
  Наслідок: ребра не перемаршрутизуються під нову позицію вузла (routing
  вже відбувся до перезапису) — прийнятно для v0, чия вимога — лише щоб
  зафіксований вузол опинився за вказаними координатами.
- Дата: 2026-07-03, коміт: (див. `phase4-step2` у progress-log.md)

## Крок 5.5 — Drill-down навігація по `details`
- Причина: план каже "Під-діаграма читається відносно шляху батьківського
  файлу" — це буквально можливо лише з доступом до файлової системи.
  Браузерний `<input type="file">`/drag&drop дає доступ лише до File-
  об'єктів, які користувач явно обрав чи перетягнув; веб-застосунок не
  може самостійно прочитати довільний відносний шлях (`./oauth-
  detail.dc.yaml`) з диска без File System Access API (Chromium-only,
  не має еквівалента в Firefox/Safari, і вимагає окремого user-gesture
  на директорію — несумісно з "подвійний клік по вузлу" як триггером).
- Рішення: `details` резолвиться проти in-memory `virtualFS` — мапи
  basename → вміст файлу, заповненої з усіх файлів, обраних РАЗОМ в
  одній дії (multi-select у file input, або drop кількох файлів/папки).
  Якщо файл, на який вказує `details`, не входить у цю вибірку — це
  трактується так само, як "битий details-шлях" з AC (нефатальна
  помилка, поточна діаграма лишається) — той самий UX, що й вимагає
  план для дійсно відсутнього файлу.
- Дата: 2026-07-03, коміт: `2fd6df3`

## Крок 6.3 — Flow-плеєр на React Flow
- Причина: старий SVG-в'ювер реверсує шлях маркера
  (`animateMotion path=...`), коли напрямок кроку flow протилежний
  напрямку самого `link` (тобто крок трактується як "response").
  `DcEdge` (React Flow custom edge) отримує лише `data.isActive` —
  без інформації про те, чи крок йде "вперед" чи "у відповідь" відносно
  `from`/`to` ребра; прокидування цього прапорця вимагало б розширення
  edge-даних інформацією про поточний активний flow-крок при кожному
  рендері (а не лише прапорцем active/visited), що на цьому кроці
  визнано зайвим ускладненням заради суто візуального нюансу.
- Рішення: маркер на React Flow-канві завжди рухається від source до
  target ребра, незалежно від напрямку кроку. Функціонально плеєр
  (підсвітка, note, вибір гілки, autoplay) не постраждав — розбіжність
  суто в напрямку анімації маркера для "response"-кроків. Якщо це
  стане проблемою для UX, розширити `DcEdgeData` полем `reversed` у
  майбутньому кроці.
- Дата: 2026-07-03, коміт: (див. `phase6-step3` у progress-log.md)

## Крок 7.2 — CRUD вузлів з канви
- Причина: додавання власного `onNodeClick` (для вибору вузла й показу
  панелі властивостей) поруч із вже наявним `onNodeDoubleClick` (drill-
  down, крок 6.4) зламало подвійний клік — не в тестовому середовищі
  специфічно, а структурно: одинарний клік синхронно оновлює React-стан
  (`selectedNodeId`), що на рівні `App`/`FlowCanvas` перераховує масив
  `nodes`, переданий у `<ReactFlow>`; React Flow ресинхронізує свій
  внутрішній store з цього нового масиву між двома фізичними кліками
  подвійного кліку. Якщо в цей момент DOM-вузол під курсором
  перестворюється, браузер не розпізнає другий клік як частину того ж
  жесту — нативна подія `dblclick` не спрацьовує (Playwright
  `.dblclick()` як і реальна миша дали однаковий результат: замість
  drill-down відбувався звичайний вибір вузла).
- Рішення: `FlowCanvas` тепер відкладає виклик `onNodeClick` на 250ms
  (`clickTimer` через `setTimeout`), а `onNodeDoubleClick` скасовує цей
  таймер, якщо він ще не спрацював. Це стандартний патерн розрізнення
  click/dblclick у UI-бібліотеках і не вимагає змін у React Flow.
  Побічний ефект: одинарний клік для виділення вузла (властивості) тепер
  має ~250ms затримку перед тим, як з'явиться панель властивостей —
  непомітно для людини, але важливо знати для будь-яких майбутніх
  playwright-тестів на клік (не dblclick): чекати з запасом понад 250ms
  або використовувати `expect(...).toBeVisible()` (яке само чекає).
- Дата: 2026-07-03, коміт: (див. `phase7-step2` у progress-log.md)

## Крок 7.4 — Редактор flows
- Причина: `applyOps` (введений у 7.2) читав базовий текст із React-стану
  `current.rawText` через замикання; коли записуєш flow клікаючи по
  кількох ребрах підряд, кожен клік викликає `applyOps` асинхронно
  (`void applyOps(...)`) без очікування завершення попереднього — а
  React-стан оновлюється не синхронно. Другий клік стартував зі
  СТАРОГО `current.rawText` (без кроку, доданого першим кліком), і
  асинхронна операція, що завершувалась пізніше, перезаписувала
  результат раніше завершеної — фактично губився крок посеред запису
  (відтворено і в реальному використанні, не лише в
  Playwright-тесті: швидкі послідовні кліки по ребрах губили проміжні
  кроки).
- Рішення: `App.tsx` тримає `levelRef` — синхронне дзеркало поточного
  рівня, яке оновлюється в той самий момент, що й `setStack` (усередині
  updater-функції, яка виконується синхронно відносно виклику), а не
  чекає рендеру; `applyOps` читає базовий текст із `levelRef.current`,
  а не з замикання. Додатково `applyChainRef` серіалізує накладені
  виклики `applyOps` (`.then(run, run)`), щоб кожен наступний патч
  застосовувався до результату попереднього, а не паралельно до того
  самого вихідного тексту.
- Дата: 2026-07-03, коміт: (див. `phase7-step4` у progress-log.md)

## Крок 7.7 — Undo/Redo (продовження гонки з кроку 7.4)
- Причина: виправлення з кроку 7.4 (`levelRef`+`applyChainRef`) лишало
  один залишковий дефект: `levelRef.current` оновлювався ВСЕРЕДИНІ
  updater-функції, переданої в `setStack`. React 18 (з `createRoot`)
  автоматично батчить `setState`-виклики і НЕ гарантує, що updater-
  функція виконається синхронно в момент виклику `setState` — вона може
  виконатись пізніше, коли React обробить батч. Коли додавання історії
  (крок 7.7) додало ще один `setState` (`setHistoryCounts`) у той самий
  потік виконання, це змінило час обробки батчу настільки, що стало
  100%-відтворюваним: другий `applyOps`-виклик (клік по другому ребру
  під час запису flow) читав `levelRef.current`, який ще не встиг
  оновитись усередині updater'а першого виклику — губився крок
  посередині запису. Діагностовано інструментальним логуванням
  (`console.log` синхронних і асинхронних меж) — показало, що
  `RUN START` другого виклику стартував ДО того, як перший виклик
  встиг виконати свій `updateCurrentLevel`.
- Рішення: `updateCurrentLevel` тепер обчислює злиття (`merged`) і
  присвоює `levelRef.current = merged` синхронно ПОЗА updater-функцією
  `setStack`, одразу в тілі `updateCurrentLevel` — до будь-якого виклику
  `setState`. Сам `setStack`-updater лише підставляє вже готовий
  `merged` на позицію останнього рівня в стеку (не залежить від `prev`
  для самого вмісту, лише для довжини стеку), тож коректність більше не
  залежить від того, коли React вирішить обробити батч.
- Дата: 2026-07-03, коміт: (див. `phase7-step7` у progress-log.md)

## Крок 8.4 — Деплой і CI
- Причина: план вимагає "Сайт доступний за публічним URL... URL
  зафіксувати в README" — репозиторій спершу не мав GitHub-remote.
  Власник підтвердив (в чаті) створення репозиторію й увімкнення Pages
  через `gh`, тож ці дії виконано з його прямого дозволу, а не
  самовільно.
- Рішення: обрано GitHub Pages (не Cloudflare Pages) — найпростіший
  варіант без додаткового акаунта, вже передбачений у плані як опція
  "вирішити при виконанні". `gh repo create oleksii-mykhniak/diagramcore`
  (спершу приватний — власник обрав це в AskUserQuestion), `git push -u
  origin main`, `gh api -X POST repos/.../pages -f build_type=workflow`
  → GitHub відхилив Pages для приватного репо (план акаунту не
  підтримує); за підтвердженням власника репозиторій переведено в
  public (`gh repo edit --visibility public`), після чого Pages
  увімкнувся. Перший `Deploy web` run впав на кроці
  `actions/deploy-pages@v4` з "Deployment failed, try again later" —
  типова тимчасова затримка провіжинінгу Pages одразу після першого
  увімкнення; `gh run rerun --failed` пройшов успішно з другої спроби.
  `.github/workflows/deploy-web.yml` (build dc + wasm → `npm run build`
  → `npx playwright test` проти прод-білда → `actions/deploy-pages`) і
  `.github/workflows/ci.yml` (той самий шлях без деплою, для PR/push)
  обидва зелені на GitHub Actions; `vite.config.ts` — `base: './'` +
  виправлені абсолютні шляхи (`wasmValidate.ts` fetch, `examples.ts`
  превью-URL) на відносні до `import.meta.env.BASE_URL` — перевірено
  вручну на живому сайті (`curl` на `/`, `/dc.wasm`,
  `/example-previews/auth-system.svg` — усі 200).
  Сайт: https://oleksii-mykhniak.github.io/diagramcore/
- Дата: 2026-07-03, коміт: (див. `phase8-step4` у progress-log.md)
