# DiagramCore — концепт

## Проблема

Діаграми архітектури (auth-flow, платіжні системи тощо) зазвичай малюються
вручну в графічних редакторах: важко тримати їх синхронними з кодом, важко
згодовувати AI-асистентам як контекст, важко версіювати в git.

## Ідея

Єдине мовно-незалежне джерело правди — YAML-файл `*.dc.yaml`, що описує:

- **nodes** — компоненти системи (actor, service, storage, queue, external, component)
- **links** — зв'язки між компонентами (request, call, query, event, dataflow, inherits, contains)
- **flows** — сценарії використання як послідовність кроків по існуючих зв'язках,
  з підтримкою умовних гілок (branch)

З цього єдиного файлу генеруються:

1. **AI-контекст** (`dc context`) — markdown-опис для вставки в чат з LLM,
   щоб асистент розумів архітектуру системи без прочитання всього коду.
2. **Валідація** (`dc validate`) — семантичні перевірки (посилання на неіснуючі
   вузли, кроки flow без відповідного link, дублікати id тощо).
3. **Рендер** (`dc render`) — SVG/PNG діаграми через D2, з підсвічуванням
   конкретного flow (статично або покроково/анімовано) для слайдів і документації.
4. **Web-редактор** (фаза 5) — інтерактивний перегляд і drag&drop правка позицій,
   flow-плеєр з анімацією руху по кроках.

## Ключові принципи

- YAML — джерело правди; layout (координати) зберігається окремо, щоб не
  забруднювати семантику розташуванням.
- Flow може ходити лише по вже оголошених зв'язках — це змушує тримати links
  і flows консистентними.
- Підтримка вкладених діаграм (`details`) — деталізація одного вузла в окремий
  файл, наприклад `oauth-detail.dc.yaml` для вузла OAuth-провайдера.
- Формат розрахований одночасно на людське читання (git diff, review) і на
  машинну обробку (JSON Schema, CI-валідація).

Детальна специфікація формату — `docs/format.md`. План реалізації — `PLAN.md`.
