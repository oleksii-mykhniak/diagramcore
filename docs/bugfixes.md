## 2026-07-05 — Ресайз вузла: фігура "відлітає" в протилежний бік по завершенні

**Симптом:** при зміні розміру вузла (особливо перетягуванням верхнього/
лівого хендла) під час драгу все виглядає коректно, але одразу по
відпусканню миші фігура стрибає назад у бік, протилежний напрямку
ресайзу — розмір лишається новим, а позиція повертається до старої.

**Причина:** React Flow `NodeResizer` під час ресайзу з верхнього/лівого
хендла коректно рухає `x`/`y` вузла (щоб протилежний край лишався на
місці), керуючи цим через власний внутрішній стан. Але `onResizeEnd`
у коді цього проєкту передавав далі лише `{width, height}`, ігноруючи
нові `x`/`y` з `params`. Канва — контрольований компонент (`FlowCanvas`
перераховує `position` з `positions[id]` на кожен рендер), тож коміт
самого лише розміру призводив до того, що на наступному рендері вузол
отримував нову ширину/висоту, але стару позицію.

**Виправлення:**
- `web/src/components/rfNodeTypes.tsx` — `onResizeEnd` тепер передає
  ще й `x`, `y` з `params` (для обох типів вузлів — leaf і container).
- `web/src/components/FlowCanvas.tsx` — конвертує ці координати (RF
  віддає їх відносно `parent` для вкладених вузлів) в абсолютні і
  передає в `onNodeResizeStop`.
- `web/src/hooks/useDiagramEditing.ts` — `onNodeResizeStop` тепер
  комітить одночасно `sizes` і `positions` (раніше — лише `sizes`),
  за тим самим патерном, що й `onNodeDrag`.
- `web/src/components/EditorWorkspace.tsx` — оновлено сигнатуру пропа.

**Перевірка:** `tsc --noEmit` чисто, `npm test` — 91/91 passed,
`npm run build` успішний.

## 2026-07-05 — На кожен push/PR у main запускалось 3 GitHub Actions замість 2

**Симптом:** на push у `main` (і на PR) паралельно стартували три
workflow-прогони: `CI`, `Deploy web` і `Diagrams`, хоча очікувались
лише перші два.

**Причина:** `.github/workflows/diagrams.example.yml` — це приклад
workflow'а для користувачів DiagramCore (описаний у `docs/ci.md`, мали
скопіювати собі під іншою назвою), а не активний CI-крок цього
репозиторію. Але GitHub Actions підхоплює як активні workflow'и всі
файли з розширенням `.yml`/`.yaml` прямо в `.github/workflows/`
незалежно від імені — тому "приклад" реально запускався в самому
DiagramCore на кожен push/PR у `main`.

**Виправлення:** файл перенесено з `.github/workflows/diagrams.example.yml`
у `docs/ci-examples/diagrams.example.yml` (поза директорією, яку сканує
GitHub Actions) — лишається чинним прикладом для копіювання, але більше
не виконується як реальний workflow тут. Оновлено посилання в
`docs/ci.md`.

## 2026-07-05 — Клік на фігуру А лишається "активним" ~1с, потім заважає перетягуванню фігури Б

**Симптом:** клацнути фігуру А, одразу після цього почати тягнути
фігуру Б — фігура А ще ~1с виглядає обраною/активною, а рух фігури Б
виглядає так, ніби щось "скасовує" його на середині жесту.

**Причина:** `onNodeClick` у `FlowCanvas.tsx` навмисно відкладає
side-effect вибору на 250мс через спільний `clickTimer` (ref), щоб
відрізнити одиночний клік від подвійного (комент у коді, PLAN.md крок
7.2) — інакше подвійний клік не спрацьовує, бо перший клік встигає
перерендерити DOM-вузол. Таймер скидається лише наступним *кліком*
(в `onNodeClick`/`onNodeDoubleClick`), але не стартом *драгу*. Тож
послідовність клік А → одразу драг Б (mousedown+move, без click-події)
лишає таймер А живим; він спрацьовує ~250мс потому, посеред драгу Б,
і викликає `setSelectedNodeId('A')`/`setSelectedNodeIds(['A'])` —
затираючи вибір, який React Flow вже нативно перемкнув на Б. Це, в
свою чергу, перезапускає ефект, що патчить `selected`/`isSelected` на
живі ноди (`FlowCanvas.tsx` — окремий ефект, щоб не чіпати `allNodes`
і не скидати позицію під час активного драгу), знімаючи виділення з Б
і повертаючи його на А — візуально невідрізнимо від "скасування" руху Б.

**Виправлення:** `web/src/components/FlowCanvas.tsx` — доданий
`onNodeDragStart`, що скидає `clickTimer` при старті будь-якого драгу:
старт драгу означає, що намір "вибрати клікнутий вузол" вже неактуальний.

**Перевірка:** `tsc --noEmit` чисто, `npm test` — 91/91 passed,
`npm run build` успішний.
