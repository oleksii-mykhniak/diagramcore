# DiagramCore — специфікація формату v0

Файл `*.dc.yaml` — єдине джерело правди для однієї діаграми. Складається з
чотирьох верхньорівневих секцій: `diagram`, `nodes`, `links`, `flows`.

## Секція `diagram`

Метадані діаграми.

| Поле         | Тип    | Обов'язкове | Default | Опис |
|--------------|--------|:-----------:|---------|------|
| `title`      | string | так         | —       | Назва діаграми. |
| `purpose`    | string | ні          | `""`    | Навіщо існує ця діаграма / система. |
| `audience`   | string | ні          | `""`    | Для кого призначено (напр. `students`, `engineers`). Довільний рядок у v0. |
| `version`    | string | ні          | `"0.1"` | Версія формату/діаграми, довільний рядок. |
| `custom_types` | list[string] | ні | `[]`   | Додаткові типи вузлів понад базовий набір (див. нижче). Реалізація перевірки — у фазі 1 (`DC003`). |

## Секція `nodes`

Список вузлів (компонентів системи).

| Поле          | Тип    | Обов'язкове | Default | Опис |
|---------------|--------|:-----------:|---------|------|
| `id`          | string | так         | —       | Унікальний ідентифікатор. Regex: `[A-Za-z][A-Za-z0-9_]*`. Дублікат → `DC001`. |
| `type`        | string | так         | —       | Один із базових типів (нижче) або значення з `custom_types`. Невідомий тип → `DC003`. |
| `label`       | string | ні          | значення `id` | Людиночитна назва, показується на діаграмі. |
| `description` | string | ні          | `""`    | Опис призначення вузла. |
| `ai_context`  | string | ні          | `""`    | Додатковий текст спеціально для генератора AI-контексту (фаза 2) — деталі, які не потрібні на візуальній діаграмі, але корисні LLM. |
| `tags`        | list[string] | ні    | `[]`    | Довільні мітки для групування/фільтрації (не впливають на валідацію в v0). |
| `details`     | string (шлях) | ні   | —       | Референс на під-діаграму, що деталізує цей вузол. Шлях **відносний до файлу, в якому оголошено `details`**. Файл має існувати і бути валідним `*.dc.yaml`, інакше → `DC006`. |

### Базові типи вузлів (v0)

`actor`, `service`, `storage`, `queue`, `external`, `component`.

Розширення набору — через `custom_types` на рівні `diagram`. Семантика
рендерингу для кастомних типів визначається у фазі 3 (мапляться на
дефолтний shape, якщо немає спеціального стилю).

## Секція `links`

Список зв'язків між вузлами.

| Поле       | Тип     | Обов'язкове | Default | Опис |
|------------|---------|:-----------:|---------|------|
| `from`     | string  | так         | —       | `id` вузла-джерела. Неіснуючий вузол → `DC002`. |
| `to`       | string  | так         | —       | `id` вузла-цілі. Неіснуючий вузол → `DC002`. |
| `type`     | string  | так         | —       | Один із: `request`, `call`, `query`, `event`, `dataflow`, `inherits`, `contains`. Невідомий тип → `DC003`. |
| `label`    | string  | ні          | `""`    | Підпис на ребрі. |
| `directed` | bool    | ні          | `true`  | Якщо `false` — ребро без напрямку (рендериться без стрілки), і flow може ходити в обидва боки без різниці. |

## Секція `flows`

Список сценаріїв використання системи. Кожен flow — послідовність кроків по
вже оголошених `links`.

| Поле    | Тип           | Обов'язкове | Default | Опис |
|---------|---------------|:-----------:|---------|------|
| `name`  | string        | так         | —       | Унікальна назва flow у межах файлу (використовується в `dc render --flow`). |
| `steps` | list[Step \| Branch] | так, непорожній | — | Порожній список → `DC007`. |

### Step (звичайний крок)

| Поле   | Тип    | Обов'язкове | Опис |
|--------|--------|:-----------:|------|
| `from` | string | так         | `id` вузла-джерела кроку. |
| `to`   | string | так         | `id` вузла-цілі кроку. |
| `note` | string | ні          | Пояснення кроку (використовується в AI-контексті та в панелі flow-плеєра). |

**Семантика валідності кроку:** крок `from → to` валідний лише якщо існує
`link` між цими вузлами (`DC004`, якщо немає; `DC005`, якщо `from`/`to`
взагалі не існують як вузли). Для `directed: true` link напрямок `from → to`
у кроці має збігатися з напрямком, оголошеним у `links` **або** бути
зворотним — прохід кроку в зворотному напрямку по directed-ребру дозволений
і трактується як "відповідь/response" (напр. `Gateway → User` по link
`User → Gateway`). Для `directed: false` напрямок кроку не має значення.

### Branch (умовна гілка)

| Поле        | Тип                | Обов'язкове | Опис |
|-------------|--------------------| :---------: |------|
| `branch.condition` | string      | так  | Людиночитний опис умови. |
| `branch.then`      | list[Step]  | так, непорожній | Кроки гілки "так". Порожньо/відсутнє → `DC008`. |
| `branch.else`      | list[Step]  | ні   | Кроки гілки "ні". Може бути відсутнім. |

Кожен елемент `steps` — або звичайний Step (має `from`/`to`), або Branch (має
ключ `branch`); наявність обох або жодного на одному рівні — помилка
структури (ловиться JSON Schema, крок 0.4).

## Референс на під-діаграму (`details`)

- Шлях у `details` — відносний до директорії файлу, в якому він вказаний
  (не до кореня репозиторію і не до поточної робочої директорії процесу).
- Під-діаграма — самостійний валідний `*.dc.yaml` файл, з власними `nodes`,
  `links`, `flows`.
- `dc context --deep` інлайнить вміст під-діаграми в AI-контекст (фаза 2).
- Циклічні референси (A деталізує B, B деталізує A) — **не помилка**.
  Будь-який обхід ланцюжка `details` (валідація, `dc context --deep`,
  drill-down у редакторі) ведеться з множиною вже відвіданих файлів
  (за канонічним абсолютним шляхом): кожен файл обробляється рівно один
  раз, повторний референс на вже відвіданий файл пропускається.

## Layout-файл (`<name>.layout.json`)

Ручні позиції вузлів — окремий файл поруч із ядром діаграми, **не**
частина `*.dc.yaml`. Дозволяє зберігати розташування, обране користувачем
у web-редакторі (фаза 5) чи обчислене автолейаутом (`dc render
--write-layout`, фаза 4), і повторно використовувати його при наступних
рендерах замість того, щоб щоразу покладатись на автоматичний layout
D2/ELK.

Ім'я файлу — `<name>.layout.json`, де `<name>` — ім'я діаграми без
розширення `.dc.yaml` (напр. для `auth-system.dc.yaml` →
`auth-system.layout.json`), у тій самій директорії. `dc render`
автоматично підхоплює такий файл, якщо він існує; шлях можна перевизначити
прапорцем `--layout-file`.

Формат:

```json
{
  "views": {
    "default": {
      "positions": {
        "User": { "x": 0, "y": 0 },
        "Gateway": { "x": 200, "y": 0 }
      }
    }
  }
}
```

- `views` — мапа: назва view → позиції для неї. У v0 актуальна лише
  `"default"` (view для повного рендеру діаграми); множинні views —
  заготовка під майбутні кастомні розкладки (напр. окремий view для
  конкретного flow), реалізація поза межами v0.
- `positions` — мапа `nodeId → {x, y}` у координатах SVG/D2-канви
  (top-left origin, ті самі одиниці, що й `top`/`left` в D2).
- Вузол, чий `id` **відсутній** у `positions`, лягає автоматичним
  layout-движком (dagre/elk) як завжди — часткові layout-файли є
  нормальним станом (напр. після ручного переміщення лише кількох
  вузлів).
- `id` у `positions`, якого **немає** серед вузлів ядра діаграми — це
  warning (виводиться, не блокує рендер), не помилка валідації: ядро
  діаграми могло змінитися після того, як layout був збережений.

JSON Schema — `schema/layout.schema.json`.

## Приклад мінімальної діаграми

```yaml
diagram:
  title: "Приклад"
  purpose: "Демонстрація формату"
  audience: students
  version: "0.1"

nodes:
  - id: User
    type: actor
    label: "Користувач"
  - id: Gateway
    type: service
    description: "Вхідна точка API"

links:
  - from: User
    to: Gateway
    type: request
    label: "HTTPS запит"

flows:
  - name: "Базовий запит"
    steps:
      - from: User
        to: Gateway
        note: "Надсилає запит"
```

## `dc-style.yaml` — конвенції для набору діаграм (PLAN.md крок 9.3)

Опційний файл `dc-style.yaml`, що лежить поруч із діаграмами, які він
регламентує (шукається в директорії конкретного `*.dc.yaml` файла — не
успадковується з батьківських директорій). Відсутність файла означає
відсутність будь-яких стильових обмежень (не помилка). Усі поля
опційні; незаданe поле = немає обмеження за цим виміром.

```yaml
allowed_types:        # білий список типів вузлів (окрім базових/custom_types)
  - actor
  - service
allowed_tags:          # білий список tags
  - internal
id_pattern: "^[A-Z][A-Za-z0-9]*$"   # regex, якому має відповідати кожен node id
require_description: true            # кожен вузол повинен мати description
max_nodes: 20                         # максимум вузлів на діаграму
theme: default                        # тема рендера (довільний рядок, для render-движка)
```

Перевірка (`dc lint --style` / MCP `lint_style`) видає коди `DS0xx`:

| Код   | Умова                                              |
|-------|-----------------------------------------------------|
| DS001 | тип вузла не в `allowed_types`                       |
| DS002 | тег вузла не в `allowed_tags`                        |
| DS003 | id вузла не відповідає `id_pattern`                  |
| DS004 | `require_description: true`, а вузол без опису       |
| DS005 | кількість вузлів перевищує `max_nodes`               |

Приклад лежить у `examples/dc-style.yaml` — усі приклади в цій
директорії йому відповідають (перевіряється тестом
`internal/style.TestExamplesConformToTheirOwnStyleGuide`).
