# PLAN3 — Фаза 11: Функціональне ядро редактора

Продовження PLAN.md і PLAN2.md (фази 0–10 виконано; архів —
`docs/plans/`, журнал виконаних фаз — `docs/plans/progress-log-phases-0-10.md`).
Прогрес фази 11 трекається у свіжому `docs/progress-log.md`.

## Контекст і мета

Після фази 10 редактор виглядає пристойно, але **функціонально кульгає**:

- Канва «дрижить» при перетягуванні: `FlowCanvas` — повністю контрольований
  React Flow, кожен `mousemove` драгу йде через
  `onNodeDrag → updateCurrentLevel → setStack` на корені App, перерендерює
  весь застосунок і перебудовує масив `rfNodes` (`FlowCanvas.tsx:189-198`,
  `useDiagramEditing.ts:250-259`). Плюс на кожен рух перестворюються
  `positions`-об'єкт і `manualPositionIds`-Set.
- YAML-панель займає низ екрана, хоча це «сирий» вигляд для просунутих правок.
- Немає resize вузлів, вкладеності (контейнерів), стилів на рівні
  екземпляра, контролю стрілок/лейблів ребер, локального автозбереження.
- Типи вузлів фактично обмежені: невідомий тип валідатор вважає помилкою,
  якщо не оголошений у `custom_types`.

Мета фази — **функціональний паритет базового редагування з draw.io/Excalidraw**
(drag/resize/стилі/контейнери/ребра) при плавному перфомансі, зберігаючи
інваріант «YAML — джерело правди, канва — проєкція».

## Рішення фази (погоджено з власником 2026-07-04; не переглядати без згоди)

- **Перфоманс — перший крок фази.** Канва переходить на внутрішній стан
  React Flow (`useNodesState`): позиції під час драгу живуть у RF, у
  document-стан комітяться лише на `onNodeDragStop`. Node-компоненти —
  `memo()`. Перф-бюджет: драг на діаграмі зі 100 вузлів без падіння нижче
  ~55fps (перевірка вручну + автоматичний лічильник рендерів у тесті).
- **Стилі — гібрид.** Стилі рівня *типу* — в YAML `custom_types`
  (розширюється: `stroke`, `strokeWidth`, `lineStyle`, `rounded`).
  Оверрайди рівня *екземпляра* (вузла/лінка), розміри вузлів і позиції
  лейблів ребер — у layout-файлі (`<name>.layout.json`). YAML лишається
  семантичним; AI-контекст не забруднюється презентацією.
- **Вкладеність — `parent: <id>` на вузлі** (плоский список, як
  `parentId` у draw.io/React Flow). Обґрунтування: 1:1 мапиться на
  контейнери D2 (`a.b.c`), `subgraph` Mermaid і `parentId` React Flow —
  властивість не губиться при переході між рендерерами. ELK підтримує
  ієрархічний layout нативно.
- **Довільні типи дозволені.** Невідомий тип — НЕ помилка валідації;
  `custom_types` стає опційним місцем для стилізації типу. (Зачіпає Go
  validator + schema + WASM.)
- **Навігація по details — вкладки + breadcrumbs.** Усі details-файли
  завантажуються разом з головною діаграмою; кожна відкрита діаграма —
  вкладка над канвою; breadcrumb показує шлях drill-down. Без плаваючих
  саб-вікон.
- **YAML-панель переїжджає у правий док** (вкладка поряд з
  Properties/Links/Flows). Нижня панель зникає.
- **Імпорт draw.io тимчасово схований** за фіче-флагом (код і тести
  лишаються, пункт меню зникає). Повернення — окремим кроком у майбутній
  фазі, коли імпорт доведемо до пуття.
- **Лейбли ребер:** лейбл — атрибут лінка в YAML (як зараз), тому «текст
  без конекту» неможливий за побудовою. Видимість лейблів — режим View
  (усі/жодного) + індивідуальний show/hide у layout-файлі. AI-контекст
  завжди включає всі лейбли незалежно від видимості.
- **Локальне автозбереження — IndexedDB** (rawText + layout, за fileName,
  з дебаунсом), відновлення при старті через пропозицію «Restore unsaved
  work?». Явний Save/Open очищає відповідний запис.

**Регресія кожного кроку:** `npm test` + `npm run build` у `web/`;
для кроків, що зачіпають Go (11.6): `go test ./...`, `go vet ./...`,
`./dc validate examples/*.dc.yaml`, `make wasm && make wasm-test`.
Наприкінці фази — повний `npm run test:e2e`.
Commit: `phase11-step<M>: <опис>` + запис у `docs/progress-log.md`.

## Крок 11.1 — Перф: плавний drag без глобального перерендеру

`FlowCanvas` переходить на некontrольований патерн React Flow:
`useNodesState`/`useEdgesState`, синхронізація з пропсів — `useEffect`
лише при структурних змінах (склад вузлів/ребер, зовнішні позиції);
під час драгу позиції живуть усередині RF. `onNodeDrag(per-move)`
прибирається з ланцюга; новий колбек `onNodeDragStop(id, pos)` комітить
позицію в level один раз. Усі node/edge-компоненти обгортаються в
`memo()`. Клік-таймер (single vs double click) перевіряється — з
мемоізованими вузлами DOM більше не churn-иться, отже затримку кліку
250мс можна прибрати, якщо dblclick стабільний.

**AC:**
- Тест (vitest): лічильник рендерів App/EditorWorkspace не росте під час
  симульованих position-changes драгу; коміт стану відбувається один раз
  на dragStop.
- Перетягнутий вузол зберігає позицію після відпускання; Save пише її в
  layout-файл (наявні e2e drag-спеки зелені).
- Notes драг працює так само (коміт на dragStop).
- Ручна перевірка: драг на `examples/` найбільшій діаграмі плавний.

## Крок 11.2 — YAML у правий док

`YamlPanel` стає четвертою вкладкою `RightDock` («YAML»); нижня панель і
`yaml-panel-toggle`/resize-handle видаляються, `yamlPanelOpen/Height`
державні пропси прибираються з App. `data-testid` самої панелі
зберігається; e2e-правки — «відкрити вкладку YAML перед взаємодією».

**AC:**
- YAML доступний через вкладку правого дока, редагування+commit працює
  (наявні спеки YamlPanel зелені після адаптації відкриття).
- Нижньої панелі немає; канва займає звільнену висоту.
- Вибір вузла, як і раніше, перемикає док на Properties.

## Крок 11.3 — Сховати імпорт draw.io + локальний автосейв

1. Пункт меню File → Import draw.io ховається за фіче-флагом
   (`web/src/featureFlags.ts`, `drawioImport: false`); код/тести importer
   не видаляються.
2. Автосейв: модуль `web/src/localAutosave.ts` (IndexedDB, ключ —
   fileName): зберігає `{rawText, positions, notePositions, renderStyle,
   savedAt}` з дебаунсом ~1с на кожну мутацію рівня. При старті без
   share-link, якщо є свіжіший за savedRawText запис — банер
   «Restore unsaved work?» (Restore/Discard). Явний Save та відкриття
   нового файлу очищають запис.

**AC:**
- Import draw.io відсутній у меню при вимкненому флагу; unit-тести
  importer досі зелені.
- Тест: мутація → (fake timers) запис в IndexedDB; повторне відкриття
  того ж fileName пропонує відновлення; Restore повертає текст і позиції;
  Discard/Save очищають.
- Перезавантаження сторінки з незбереженими змінами не втрачає роботу
  (e2e-спека).

## Крок 11.4 — Resize вузлів

`NodeResizer` (@xyflow/react) на всіх dc-вузлах для виділеного вузла;
розмір пишеться в layout-файл (`views.default.sizes[id] = {w,h}`) на
resize-stop. `NodeShell` рендерить shape на фактичний розмір (у
`renderSvgInner` вже параметризовані w/h), текст переносить/обрізає.
Експорт SVG/PNG і share-link враховують розміри. Мінімальний розмір —
базовий 160×60 ділити на 2.

**AC:**
- Вузол ресайзиться хендлами як у draw.io; розмір переживає
  Save → Open (layout-файл) і share-link.
- Канва та SVG-експорт малюють однаковий розмір (юніт на svgExport).
- Auto-layout (Re-layout) резервує фактичні розміри вузлів, не базові.

## Крок 11.5 — Формат: `parent:` + довільні типи (Go core + schema + WASM)

1. `nodes[].parent: <id>` (опційний): parser/model/schema; валідація —
   parent існує, циклів немає (`a.parent=b, b.parent=a` — помилка).
2. Невідомий тип вузла більше не помилка (прибрати/послабити відповідне
   правило validator; `custom_types` лишається для стилізації).
3. `custom_types` розширюється полями `stroke`, `strokeWidth`,
   `lineStyle` (`solid|dashed|dotted`), `rounded` (bool) — назад-сумісно.
4. Рендерери: D2 — вкладені контейнери (`parent.child`), Mermaid —
   `subgraph`; AI-контекст (`dc context`) показує ієрархію (indent або
   `parent:` поле).
5. Web: types.ts/parseDiagram/yamlPatch (`updateNode` вміє parent),
   WASM перебілджений.

**AC:**
- `./dc validate` приймає діаграму з parent і довільними типами;
  ловить неіснуючий parent і цикл (юніт-тести Go).
- `dc render --to d2|mermaid` дає вкладені контейнери (golden-тести).
- `dc context` включає ієрархію (golden).
- `make wasm && make wasm-test` зелені; web-валідатор показує ті самі
  помилки.

## Крок 11.6 — Контейнери на канві

Вузли з дітьми рендеряться контейнером: React Flow `parentId` +
`extent: 'parent'`, окремий вигляд (заголовок зверху зліва, напівпрозорий
фон, за z-порядком під дітьми). Drag дитини всередині контейнера — вільний;
drag вузла В контейнер / З контейнера змінює `parent:` у YAML (патч на
dragStop за перетином меж). Resize контейнера (з кроку 11.4) не менший за
bbox дітей. Auto-layout — ієрархічний ELK (children у elk-графі);
позиції дітей у layout-файлі — відносні до контейнера.

**AC:**
- Приклад «GCP → k8s → namespace → services» (новий
  `examples/nested.dc.yaml`) рендериться вкладеними контейнерами; діти
  тягаються всередині; винесення дитини за межі знімає parent (видно в
  YAML-панелі).
- Re-layout розкладає ієрархію без перекриттів.
- SVG-експорт малює контейнери так само, як канва.
- Перф не деградує: драг контейнера з 10 дітьми плавний (діти їдуть
  разом без пер-кадрових комітів).

## Крок 11.7 — Вкладки діаграм + breadcrumbs, автозавантаження details

`useDiagramStack` рефакториться зі «стека» в «мапу відкритих рівнів +
активна вкладка»: при відкритті головного файлу всі досяжні
`details:`-файли з virtualFS парсяться одразу (помилка окремого файлу не
валить головний — показується на його вкладці). Смуга вкладок над
канвою (назва файлу, unsaved-маркер, закриття не-головних), breadcrumb
поточного шляху drill-down. Double-click по вузлу з details активує/
відкриває вкладку. Undo/history, flow-стан, автосейв — на рівень (вже
так). Стара поведінка `goToLevel` зникає.

**AC:**
- Відкриття прикладу з details показує вкладки всіх рівнів; перемикання
  миттєве (без повторного парсингу); правки в кожній вкладці незалежні.
- Breadcrumb відображає шлях і клікабельний.
- Двоклік по вузлу з details перемикає на відповідну вкладку (e2e).
- Автосейв працює на кожну вкладку окремо.

## Крок 11.8 — Стилі вузлів: інстанс-оверрайди + Properties UI

Layout-файл: `views.default.styles[nodeId] = {fill?, stroke?,
strokeWidth?, lineStyle?, rounded?}`. Пріоритет: інстанс → custom_type →
дефолт теми. PropertiesPanel отримує секцію Style: color-picker (fill,
stroke), товщина (1–4), тип лінії (solid/dashed/dotted), заокруглення
(toggle/radius). Рендеринг — через спільний shape-реєстр, отже канва,
SVG-експорт і share-link однакові. Кнопка «Reset style» знімає оверрайд.

**AC:**
- Зміна кольору/товщини/типу лінії/заокруглення видима одразу і
  переживає Save → Open та share-link.
- YAML при цьому не змінюється (тест: rawText незмінний після стилізації).
- SVG-експорт відтворює стилі (юніт-голден).
- Sketch-пресет (roughjs) поважає кольори/товщину оверрайдів.

## Крок 11.9 — Ребра: стрілки, стиль ліній, рухомі лейбли, видимість

1. Стиль ребра (layout: `views.default.edgeStyles[linkKey]`): маркери
   обох кінців (`none|arrow|open-arrow`), `lineStyle`, `strokeWidth`,
   `color`. UI — у LinksPanel + клік по ребру відкриває його властивості
   в доці.
2. Лейбл ребра можна перетягнути (offset у layout:
   `edgeLabelOffsets[linkKey]`); подвійний клік по лейблу — інлайн-правка
   тексту (патчить YAML `label`).
3. Видимість: View → «Connection labels» (show all / hide all) +
   індивідуальний show/hide (layout). `dc context` та AI-експорт завжди
   містять усі лейбли (тест).

**AC:**
- Маркери/стиль/колір ребра змінюються з UI, переживають Save → Open,
  однакові на канві та в SVG-експорті.
- Лейбл тягається незалежно від ребра і лишається на місці після
  перезавантаження; подвійний клік редагує текст (YAML міняється).
- Режим приховання лейблів не впливає на `dc context` (юніт Go/web).

## Крок 11.10 — Мультивиділення і базові операції редагування

Selection box (shift+drag / rubber band RF), спільне перетягування
виділеного, Delete видаляє виділені вузли+залежні лінки (з підтвердженням
як зараз), Cmd/Ctrl+D — дублювання вузла (новий id, ті ж type/label/стилі,
позиція зі зсувом). Esc знімає виділення.

**AC:**
- Rubber-band виділяє кілька вузлів; групове перетягування комітить всі
  позиції на dragStop одним апдейтом.
- Delete/Duplicate працюють з клавіатури і через меню Edit (e2e).
- Undo відкатує групове видалення одним кроком.

# Правила для агента-валідатора

1. Прогнати всі AC кроку як чекліст; AC з командами виконувати буквально
   і звіряти exit code/вивід.
2. Регресія: перед закриттям кроку — `go test ./...`, `go vet ./...`,
   `./dc validate examples/*.dc.yaml`; у web/ — `npm test` + `npm run build`.

4. Розбіжність плану з реальністю — не мовчки обходити: зафіксувати у
   `docs/deviations.md` (крок, причина, рішення) і продовжити.
5. Питання, що змінюють зафіксовані рішення (розділ «Рішення фази» тут і
   «Загальні рішення» в `docs/plans/PLAN.md`) — ескалювати власнику.
