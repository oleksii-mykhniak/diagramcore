# PLAN4 — Фаза 12: Продакшн-рівень редагування

Продовження PLAN.md, PLAN2.md і PLAN3.md (фази 0–11 виконано; архів —
`docs/plans/`, журнали — `docs/plans/progress-log-phases-0-10.md` та
`docs/plans/progress-log-phase-11.md`).
Прогрес фази 12 трекається у свіжому `docs/progress-log.md`.

## Контекст і мета

Фаза 11 дала функціональне ядро (контейнери, resize, інстанс-стилі,
вкладки, автосейв, мультивиділення). Але до «повноцінного механізму для
малювання будь-яких блок-схем продакшн-рівня» бракує:

- **Два відомі баги:** (а) додавання вузла з палітри «стрибає» всією
  діаграмою — `applyOps` (`useDiagramEditing.ts:62`) на КОЖНУ структурну
  правку перераховує ELK-layout, і всі вузли без ручної позиції
  (`manualPositionIds`) отримують нові авто-координати; (б) стрілки на
  ребрах відображаються неконсистентно — канва і SVG-експорт малюють
  дефолтний маркер по-різному (свідомо відкладено в кроці 11.9, див.
  `docs/deviations.md`).
- Текст не редагується на місці: лейбл вузла — тільки через Properties,
  лейбл ребра — через `window.prompt`. Немає налаштувань тексту (розмір,
  жирність, колір, вирівнювання).
- Конекти не можна приховати (тільки їх лейбли); немає режиму «показати
  все приховане», щоб бачити повний склад ядра.
- Немає z-порядку (шарів), групування виділеного, власних картинок,
  copy/paste, вирівнювання/розподілу.
- Вкладки дока Properties і Links дублюють одна одну — клік по ребру та
  клік по вузлу ведуть у різні панелі.

Мета фази — **закрити розрив до продакшн-редактора блок-схем**: усунути
обидва баги, дати повний цикл роботи з текстом, приховування з
«рентген-режимом», шари, групування, картинки, copy/paste і
вирівнювання — зберігаючи інваріант «YAML — джерело правди (і те, що
бачить AI), канва — проєкція, презентація — в layout-файлі».

Анімація (експорт GIF + WebM) — окрема наступна фаза (PLAN5), ескіз —
наприкінці цього файлу.

## Рішення фази (погоджено з власником 2026-07-18; не переглядати без згоди)

- **«Links у property» = об'єднання панелей, НЕ зміна формату.** Вкладка
  Links зливається з Properties: панель контекстна (вибраний вузол →
  властивості вузла; вибране ребро → властивості лінка; нічого не
  вибрано — огляд діаграми). YAML-формат не чіпається.
- **Приховування — тільки презентація.** Приховані конекти/тексти
  живуть у layout-файлі (`hiddenEdges`, `hiddenNodeLabels`; наявний
  `hiddenEdgeLabels` — з фази 11). YAML і `dc context` (AI-контекст)
  завжди містять усе — приховування структурно не може «загубити» дані.
- **Режим «показати приховане» (Core view)** — View-тумблер: усі
  приховані елементи тимчасово видимі з візуальною позначкою
  (напівпрозорість + бейдж), щоб точно бачити, з чого складається ядро.
- **Налаштування тексту — інстанс-стилі layout-файлу** (розширення
  `styles`/`edgeStyles` полем `text{...}`), як і всі презентаційні
  оверрайди фази 11.
- **Шари — `zOrder` у layout-файлі** (порядок малювання). Контейнер
  завжди під своїми дітьми незалежно від zOrder.
- **Картинки — окремими файлами поруч із YAML** (тека `assets/`),
  у layout-файлі — лише відносний шлях. При SVG/PNG-експорті картинка
  інлайниться як data-URI, щоб експортований файл був self-contained.
- **Групування — через наявний механізм `parent:`/контейнерів** (Ctrl+G
  створює контейнер-вузол над виділеним, Ctrl+Shift+G розгруповує) — без
  нового поняття «група» у форматі.
- **Структурні правки не рухають наявні вузли.** Після фікса 12.1
  авто-layout запускається ТІЛЬКИ явно (кнопки Re-layout / Re-layout
  all) і при першому відкритті без layout-файлу.
- **Анімаційний експорт (фаза 13, PLAN5): і WebM, і GIF** — один
  механізм покадрового рендеру, два енкодери.

**Регресія кожного кроку:** `npm test` + `npm run build` у `web/`;
для кроків, що зачіпають Go (12.4, 12.6, 12.8, 12.9): `go test ./...`,
`go vet ./...`, `./dc validate examples/*.dc.yaml`, а якщо зачеплено
validator/WASM — `make wasm && make wasm-test`.
Наприкінці фази — повний `npx playwright test`.
Commit: `phase12-step<M>: <опис>` + запис у `docs/progress-log.md`.

## Крок 12.1 — Баг: додавання елемента «стрибає» всією діаграмою

`applyOps` перестає перекладати наявні вузли: позиції ВСІХ вузлів, що
існували до правки, зберігаються як були (незалежно від
`manualPositionIds`); авто-координати з `computeLayout` отримують лише
вузли, яких раніше не було (drop з палітри і так дає їм
`manualPositions`, тож ELK-перерахунок на структурну правку або
прибирається зовсім, або його результат застосовується тільки до
новостворених id). Явні Re-layout / Re-layout all працюють як раніше.
Перевірити той самий симптом і для `applyTextReplace` (правка YAML у
доці) — там перерахунок для вузлів, що не змінились, теж не має рухати
канву.

**AC:**
- e2e: відкрити приклад, перетягнути один вузол, додати вузол з палітри
  drag-n-drop'ом — позиція ЖОДНОГО наявного вузла не змінилась (і
  ручна, і авто); новий вузол стоїть у точці дропу.
- e2e: видалення вузла не рухає решту; undo повертає і вузол, і позиції.
- Re-layout all досі перерозкладає все; наявні спеки drag/containers
  зелені.

## Крок 12.3 — Інлайн-редагування тексту на канві

   текст видаляє `label` з лінка.
4. Врахувати відому пастку 250мс click/dblclick таймера і

## Крок 12.5 — Об'єднання Links у Properties (контекстна панель)

`RightDock` стає: **Properties / Flows / YAML**. Панель Properties
контекстна:

- вибраний вузол → як зараз (поля вузла + Style + Text);
- вибране ребро (клік по ребру на канві) → властивості лінка:
  from/to/type/label + Style/Text ребра + видимість (детально — крок
  12.6) + кнопка видалення лінка;
- нічого не вибрано → огляд діаграми: назва/опис + компактний список
  вузлів і лінків (клік — вибрати відповідний елемент; це зберігає
  можливість дістатись будь-якого лінка без кліку по канві).

`LinksPanel.tsx` не видаляється в нікуди — його форма редагування лінка
переїжджає в Properties як секція. Всі e2e, що відкривали вкладку
Links, адаптуються.

**AC:**
- Вкладки дока: Properties/Flows/YAML (Links немає); клік по ребру
  відкриває Properties з формою лінка; клік по вузлу — з формою вузла.
- Порожнє виділення показує огляд; клік по лінку в списку виділяє його
  на канві і розгортає форму.
- Все, що вміла стара LinksPanel (маркери, стиль, лейбл, hide label,
  reset), доступне в новій секції; спеки edge-style/links адаптовані й
  зелені.

## Крок 12.7 — Core view: режим «показати все приховане»

View-меню: тумблер «Core view» (`useViewSettings`, localStorage, як
grid/snap). Увімкнений: усі приховані ребра/лейбли (з 12.6 і
`hiddenEdgeLabels`) відображаються напівпрозоро (напр. opacity 0.35,
пунктир для прихованих ребер) з маленьким бейджем-оком; елементи в
цьому режимі клікабельні — можна вибрати і зняти прихованість без
пошуку в списку. Вимкнення повертає звичайний вигляд. SVG-експорт
ІГНОРУЄ Core view (експортується звичайний вигляд) — це режим
перегляду, не презентації.

**AC:**
- e2e: сховати конект і лейбл → увімкнути Core view → обидва видимі
  напівпрозоро з позначкою; клік по прихованому ребру відкриває його
  властивості; зняти галочку Hide → елемент стає звичайним ще в
  Core view; вимкнути режим → нескасовані приховання знову зникли.
- SVG-експорт при ввімкненому Core view не містить прихованих
  елементів.

## Крок 12.9 — Власні картинки

1. Layout-файл: `styles[id].image: "<відносний шлях>"` (тека
   `assets/` поруч із YAML; шлях відносно YAML-файлу). Go + schema +
   web-дзеркало.
2. Завантаження: кнопка «Image…» у властивостях вузла → file picker;
   при роботі через File System Access API (`nativeFile.ts`) файл
   копіюється в `assets/` поруч із YAML; при роботі без FS API
   (virtualFS/drag-n-drop) — зберігається в virtualFS і потрапляє в
   zip при «Save as zip». Прев'ю і кнопка Remove у панелі.
3. Канва: вузол із image малює картинку всередині фігури (object-fit

4. SVG/PNG-експорт: картинка інлайниться як data-URI (self-contained
   експорт). Розумний ліміт розміру файлу (напр. ≤2МБ) з помилкою-
   тостом при перевищенні.

**AC:**
- e2e: додати PNG до вузла → видно на канві; виживає Save→Open (шлях у
  layout-файлі, файл в assets/); Remove прибирає.

- Відкриття діаграми з layout-посиланням на відсутній файл картинки не

## Крок 12.10 — Групування виділеного

Ctrl/Cmd+G на мультивиділенні (≥2 вузлів одного рівня вкладеності):
через `applyOps` створюється новий вузол-контейнер (id `group-N`, тип
`group`), виділені вузли отримують `parent:` на нього — далі працює вся
наявна механіка контейнерів (drag разом, resize, reparent). Bbox
контейнера рахується з дітей. Ctrl/Cmd+Shift+G на вибраному
контейнері-групі: діти отримують `parent:` контейнера-групи (або
знімають, якщо група top-level), сам вузол групи видаляється; позиції
дітей не змінюються. Пункти Group/Ungroup в Edit-меню і контекстному
меню. Один крок undo на операцію.

**AC:**
- e2e: rubber-band 3 вузлів → Ctrl+G → контейнер охоплює їх, драг
  контейнера тягне всіх; YAML має `parent:` і вузол групи.
- e2e: Ungroup повертає все як було (позиції не стрибнули — інваріант
  кроку 12.1); undo після Group скасовує все одним кроком.
- Групування дітей різних батьків заблоковане (пункт меню disabled).

## Крок 12.11 — Copy/paste + вирівнювання/розподіл

1. Cmd/Ctrl+C/V: копіює виділені вузли (з їхніми стилями, розмірами і
   лінками МІЖ скопійованими вузлами) у внутрішній буфер; вставка — зі
   зсувом +40/+40, нові унікальні id (той самий генератор, що
   Duplicate), працює і на іншій вкладці (стилі/розміри їдуть у
   layout-стан цільової вкладки). Cmd/Ctrl+X = копія + видалення.
2. Вирівнювання для мультивиділення (Edit-меню + контекстне меню):
   Align left/center/right/top/middle/bottom; Distribute
   horizontally/vertically. Один коміт позицій на операцію (як
   груповий drag 11.10).

**AC:**
- e2e: копіювання 2 з'єднаних вузлів → вставка дає 2 нові вузли з
  лінком між ними, стилі збережені; вставка на іншій вкладці працює;
  undo одним кроком.
- e2e: Align top для 3 вузлів вирівнює y; Distribute horizontally дає
  рівні проміжки; undo повертає позиції одним кроком.
- Шорткати не спрацьовують у текстових полях (той самий guard, що
  11.10).

# Правила для агента-валідатора

1. Прогнати всі AC кроку як чекліст; AC з командами виконувати буквально
   і звіряти exit code/вивід.
2. Регресія: перед закриттям кроку — `go test ./...`, `go vet ./...`,
   `./dc validate examples/*.dc.yaml`; у web/ — `npm test` + `npm run build`;
   якщо крок зачіпає Go/WASM — ще `make wasm && make wasm-test`.
3. Крок не закривається, якщо для нового коду немає тестів, передбачених AC.
4. Розбіжність плану з реальністю — не мовчки обходити: зафіксувати у
   `docs/deviations.md` (крок, причина, рішення) і продовжити.
5. Питання, що змінюють зафіксовані рішення (розділ «Рішення фази» тут і
   «Загальні рішення» в `docs/plans/PLAN.md`) — ескалювати власнику.
